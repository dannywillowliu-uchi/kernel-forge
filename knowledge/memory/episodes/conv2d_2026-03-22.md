# Episode: Conv2d v2 (GPU MODE conv2d_v2 leaderboard)

## Result
Leaderboard: 6.48ms mean, 6.47ms best (B200 on Modal)
Baseline: 51.9ms (F.conv2d without benchmark=True)
Speedup: 8x over naive baseline
Theoretical FP32 minimum: 3.53ms (1.7 TFLOPS / 481 TFLOPS peak)
Efficiency: 55% of FP32 peak

## Shape
size=256, batch=1, channels=128, kernelsize=32
Input: [1, 128, 256, 256], Kernel: [128, 128, 32, 32], Output: [1, 128, 225, 225]
GEMM equivalent: M=128, K=131072, N=50625 (1.7 TFLOPS)

## Winning Approach
Simple F.conv2d with `cudnn.benchmark=True` and `allow_tf32=False`:
```python
torch.backends.cudnn.allow_tf32 = False
torch.backends.cudnn.benchmark = True
return F.conv2d(input_tensor, kernel, stride=1, padding=0)
```

## Key Constraint
Reference uses DeterministicContext which disables TF32 and enables deterministic mode.
Tolerance is rtol=1e-3, atol=1e-3. TF32 produces ~0.02 max error which fails for many elements.
Any algorithm that doesn't exactly match the reference's FP32 computation order fails correctness.

## Approaches Tried
1. F.conv2d default (TF32 on by default) -> 51.9ms on Modal, FAILS correctness
2. F.conv2d with TF32 off, benchmark=False -> 51.9ms on Modal, correct but slow
3. **F.conv2d with TF32 off, benchmark=True -> 6.48ms on Modal, correct -- BEST**
4. torch.compile max-autotune -> 6.72ms (dispatch overhead)
5. cuDNN C API with FindAlgorithm -> 6.48ms but FAILS correctness (picks different algo)
6. cuDNN C API with FindAlgorithmEx -> same issue, different GEMM accumulation order
7. cuDNN v9 graph API (NCHW, heur A) -> 6.48ms, correct
8. cuDNN v9 graph API (NHWC) -> 6.48ms, correct but layout conversion overhead
9. cuDNN TENSOR_OP_MATH_ALLOW_CONVERSION -> uses TF32 internally, FAILS correctness
10. channels_last F.conv2d -> 6.70ms, slower due to conversion
11. im2col + cuBLAS SGEMM -> 224ms (26.5 GB column matrix, memory-bound)
12. Tiled im2col + cuBLAS -> 90ms, FAILS correctness (different accumulation order)
13. Grouped im2col (split input channels) -> 91ms, FAILS correctness
14. FFT convolution (float32 and float64) -> max_diff=0.03, FAILS correctness
15. CUDA graph capture -> 6.53ms (copy overhead negates launch savings)
16. Split output channels -> 6.61ms (kernel launch overhead)
17. Pre-warming cuDNN at import -> no improvement (eval already warms up)

## Key Insights
1. For 32x32 kernels, cuDNN on CUDA 12.8 uses Winograd nonfused (5.5ms locally but 129ms implicit GEMM fallback). On CUDA 13.0 (Modal), benchmark=True finds a much faster algorithm (6.48ms).
2. The massive speedup from benchmark=True (51.9ms -> 6.48ms = 8x) comes entirely from cuDNN's algorithm autotuner finding a faster internal kernel. Without benchmark=True, PyTorch's heuristic picks a slow algorithm.
3. FFT-based convolution gives 0.03 max error regardless of FFT precision (float32 or float64). The error comes from different computation order, not FFT rounding.
4. im2col for this shape materializes 26.5 GB of data -- fundamentally memory-bound.
5. cuDNN C API's FindAlgorithmEx picks algorithms that don't match PyTorch's reference output. PyTorch adds its own algorithm filtering for correctness.
6. All cuDNN APIs (legacy, v9 graph, PyTorch wrapper) converge to the same 6.48ms kernel time. The bottleneck is the CUDA kernel itself.
7. The 45% efficiency gap (6.48ms vs 3.53ms theoretical) comes from the implicit GEMM's irregular memory access pattern for a 32x32 kernel.

## What Would Help
1. A custom CUTLASS implicit GEMM kernel tuned for M=128, K=131072, N=50625 with 32x32 spatial tiling
2. A hybrid algorithm that combines FFT (for speed) with a correction pass (for accuracy)
3. Better cuDNN version -- CUDA 13.0 already improved over 12.8, future versions may improve further
