# Episode: TriMul (Triangle Multiplicative Update)

## Result
Best: 0.762ms geomean (research-first agent), 0.806ms stable (torch.compile)
Speedup: 9.25x over PyTorch reference
Hardware utilization: ~36% TF32 (memory-bandwidth bound)

## Approaches tried (in order)
1. torch.compile max-autotune-no-cudagraphs -> 0.786ms -> worked well, fuses elementwise chain
2. bf16 fused 5 projections into 1 GEMM -> 7.2ms to 2.4ms -> huge win, reduces kernel launches
3. coordinate_descent_tuning + dynamic=False -> 0.89ms to 0.86ms -> minor
4. Custom Triton gate+permute kernel (16 variants) -> all slower -> torch.compile already fuses this
5. CUDA C++ kernel -> 0.026ms slower -> Triton compiler generates better code
6. CUDA graphs -> 0.91ms -> shape recompilation overhead
7. bf16 LayerNorm -> fails correctness on cauchy distribution
8. cuEquivariance drop-in -> architecture mismatch (gate ordering differs)
9. cuEquivariance components (fused_sigmoid_gated_dual_gemm) -> 2.0ms -> individual ops fast but total slower than torch.compile
10. Liger LayerNorm -> 2.5x faster in isolation but eager pipeline slower overall

## Key insight
torch.compile's ceiling comes from inability to fuse across matmul boundaries. Production implementations (cuEquivariance, TTT-Discover) use 4 custom fused mega-kernels. But cuEquivariance's architecture doesn't match our reference (different gate ordering), so it can't be used as a drop-in. Beating torch.compile requires writing the full fused pipeline from scratch.

## What would I do differently
1. Research production implementations FIRST (cuEquivariance, TTT-Discover paper)
2. Check if the architecture matches before trying to use library kernels
3. Accept that torch.compile at 0.78ms is the practical ceiling without writing custom CUDA for the full pipeline
4. Focus time on the fused LN+projection and LN+gate+projection custom Triton kernels
